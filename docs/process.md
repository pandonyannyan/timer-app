# Timer App — Progress

## Назначение документа

Документ фиксирует текущий рабочий статус проекта, ближайшие этапы и чек-лист проверки после изменений.

Архитектурные решения, модели данных, роли, статусы, планируемая БД и API описаны в `docs/architecture.md`.

---

## Текущий статус

На текущем этапе реализована основная frontend-логика приложения.

Backend пока не подключён. Таймеры работают через mock-реализацию в `src/services/timerService.ts`, а Pinia store использует этот сервис как промежуточный слой перед будущим API.

Пользовательские настройки частично сохраняются в `localStorage` через отдельные services/utilities.

Frontend service layer подготовлен для последующей замены mock/localStorage-реализаций на API и БД.

Реализовано:

- фильтр “Все”;

  2. `warning`;

### Роли и права

Реализовано на уровне UI:

- `admin`;
- `manager`;
- `member`.

Текущие правила:

- `admin` может выполнять все глобальные действия и управлять пользователями;
- `manager` может создавать, редактировать, удалять, запускать и останавливать таймеры;
- `member` может запускать и останавливать таймеры;
- звук, поиск, фильтры, закрепления и порядок закреплённых таймеров считаются персональными действиями.

Важно: текущие проверки прав не являются безопасностью. После подключения backend ограничения нужно продублировать на backend и в Supabase RLS.

Реализовано:

## Временные frontend-only решения

### Mock-таймеры

Таймеры пока хранятся в `timerService`.

После перезагрузки страницы:

- добавленные таймеры исчезают;
- отредактированные таймеры возвращаются к mock-значениям;
- удалённые таймеры возвращаются.

### Изображения

Изображения сейчас отображаются через:

```ts
URL.createObjectURL(file)
```

Ограничения:

- изображение доступно только в текущей сессии браузера;
- после перезагрузки страницы изображение не сохраняется.

Позже изображения нужно перенести в Supabase Storage.

### Настройки звука

Настройки звука пока сохраняются в `localStorage`.

Позже их можно хранить в БД как пользовательские настройки.

### Закрепления

Закрепления и порядок закреплённых таймеров пока сохраняются в `localStorage`.

Позже их нужно перенести в таблицу `user_pinned_timers`.

### Роли

Роль пользователя сейчас моковая.

Позже роль должна приходить из auth/profile, а права должны проверяться на backend и в Supabase RLS.

## План выхода в MVP

Цель MVP — заменить текущие frontend-only моки реальной инфраструктурой, сохранив уже реализованную frontend-логику.

### Этап 1. Frontend service layer

Статус: выполнено.

Реализованы:

- `timerService`;
- `pinnedTimerService`;
- `soundSettingsService`.

### Этап 2. Подготовить Supabase

Нужно создать Supabase-проект и подготовить:

- Supabase Postgres;
- Supabase Auth;
- Supabase Storage;
- Supabase Realtime.

Минимальные таблицы:

- `profiles`;
- `timers`;
- `user_pinned_timers`;
- `timer_logs`.

Дополнительные таблицы:

- `user_settings`;
- `timer_user_settings`.

### Этап 3. Подготовить backend skeleton на FastAPI

Backend должен отвечать за:

- проверку авторизации;
- проверку ролей;
- валидацию данных;
- работу с таймерами;
- работу с закреплениями;
- запись логов;
- загрузку или обработку изображений.

### Этап 4. Реализовать API таймеров

Минимальные endpoints:

- `GET /timers`;
- `POST /timers`;
- `PATCH /timers/{timer_id}`;
- `DELETE /timers/{timer_id}`;
- `POST /timers/{timer_id}/restart`;
- `POST /timers/{timer_id}/stop`.

Рекомендуемый порядок подключения frontend:

1. `GET /timers`;
2. `POST /timers/{timer_id}/restart`;
3. `POST /timers/{timer_id}/stop`;
4. `POST /timers`;
5. `PATCH /timers/{timer_id}`;
6. `DELETE /timers/{timer_id}`.

### Этап 5. Реализовать авторизацию и роли

Минимальный MVP:

- вход по email/password;
- получение текущего пользователя;
- получение профиля пользователя;
- роль пользователя: `admin`, `manager`, `member`;
- защита frontend routes;
- проверка прав на backend.

### Этап 6. Перенести закрепления в БД

Endpoints:

- `GET /me/pinned-timers`;
- `POST /me/pinned-timers`;
- `DELETE /me/pinned-timers/{timer_id}`;
- `PATCH /me/pinned-timers/reorder`.

### Этап 7. Реализовать изображения таймеров

Для MVP использовать Supabase Storage.

Логика:

1. пользователь выбирает изображение;
2. изображение загружается в storage;
3. в таймере сохраняется постоянный `image_url`;
4. при редактировании можно заменить изображение;
5. при удалении изображения `image_url` очищается.

### Этап 8. Добавить realtime-обновления

Для MVP realtime нужен только для общих таймеров.

События:

- timer created;
- timer updated;
- timer restarted;
- timer stopped;
- timer deleted.

Realtime для закреплений не обязателен.

### Этап 9. Добавить логи

Логировать:

- created;
- updated;
- restarted;
- stopped;
- deleted;
- role_changed.

Не логировать:

- warning;
- signal;
- completed;
- sound enabled / disabled;
- global sound enabled / disabled;
- pinned;
- unpinned;
- pinned reordered;
- search;
- filter.

### Этап 10. Деплой

Варианты для MVP:

| Часть | Вариант |
|---|---|
| Frontend | GitHub Pages / Vercel / Netlify |
| Backend | Render / Railway / Fly.io |
| DB/Auth/Storage/Realtime | Supabase |

Нужно настроить:

- production env variables;
- backend URL для frontend;
- CORS;
- Supabase keys;
- HTTPS;
- production build.
