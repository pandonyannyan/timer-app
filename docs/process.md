# Timer App — Progress

## Проект

Многопользовательское веб-приложение для управления таймерами внутри команды.

Основная идея:

- команда работает с общим списком таймеров;
- таймеры можно запускать, останавливать, перезапускать и завершать локально;
- часть глобальных действий доступна только пользователям с определённой ролью;
- каждый пользователь может закреплять важные для себя таймеры и менять порядок закреплённых;
- в будущем данные будут храниться в БД и синхронизироваться между пользователями.

Стек:

- Frontend: Vue 3 + Vite + TypeScript + Pinia
- Backend: планируется Python + FastAPI + Supabase
- Хостинг: GitHub Pages + бесплатные сервисы

Подробная архитектура, модели данных, роли, Pinia store и будущая backend/Supabase-структура описаны в [architecture.md](./architecture.md).

## Основные компоненты

### TimersPage.vue

Главная страница таймеров.

Отвечает за:

- подключение TimersToolbar;
- подключение TimersTable;
- получение таймеров из Pinia;
- поиск;
- фильтр по статусу;
- расчёт view-статуса;
- расчёт оставшегося времени;
- разделение таймеров на закреплённые и незакреплённые;
- автоматический порядок незакреплённых таймеров;
- режим перестановки закреплённых таймеров;
- draft-порядок закреплённых таймеров при drag-and-drop.

Важная логика:

- закреплённые таймеры всегда отображаются сверху;
- закреплённые таймеры отображаются в пользовательском порядке;
- порядок закреплённых таймеров хранится в pinnedTimerIds;
- незакреплённые таймеры отображаются по view-статусу:
  - signal;
  - active;
  - stopped;
  - completed;
- внутри одного view-статуса незакреплённые таймеры сортируются по оставшемуся времени;
- если время одинаковое, используется сортировка по названию как fallback;
- в режиме перестановки отображаются только закреплённые таймеры;
- в режиме перестановки поиск и фильтр сбрасываются и блокируются.

### AddTimerModal.vue

Универсальная модалка для создания и редактирования таймера.

Режимы:

- create
- edit

В режиме создания:

- поля пустые;
- после сохранения создаётся новый мок-таймер в Pinia.

В режиме редактирования:

- поля заполнены данными выбранного таймера;
- можно изменить:
  - название;
  - описание;
  - длительность;
  - изображение;
- можно удалить текущее изображение;
- после сохранения изменения применяются в Pinia.

Валидация:

- название обязательно;
- длительность обязательна;
- длительность должна быть целым числом минут;
- длительность должна быть от 1 минуты до 24 часов.

Изображение:

- если у таймера уже есть изображение, в режиме редактирования показывается кнопка удаления изображения;
- кнопка удаления использует тот же IconButton и deleteIcon, что и кнопка удаления таймера в TimerRow.vue;
- кнопка удаления расположена по правому краю строки выбора файла;
- удаление изображения не применяется сразу, а только помечается через removeImage;
- если пользователь нажмёт “Отмена”, изображение останется без изменений;
- если пользователь нажмёт “Сохранить”, updateTimer очистит imageUrl;
- если после удаления пользователь выберет новый файл, удаление отменяется и сохраняется новый файл.

Дополнительно:

- Esc закрывает модалку.

Важно:

- выбранное изображение сейчас превращается в URL.createObjectURL(file);
- это временное решение для фронта;
- после перезагрузки страницы такая картинка не сохранится;
- в будущем изображение нужно будет загружать в Supabase Storage или другой storage.

## План выхода в MVP

Цель MVP — заменить текущие frontend-only моки реальной инфраструктурой, но не ломать уже реализованную frontend-логику таймеров.

Основной подход:

- сначала отделить Pinia store от источника данных через service layer;
- затем подключить backend и БД;
- после этого постепенно заменить mock-реализацию на API;
- локальную frontend-логику статусов `signal` и `completed` сохранить;
- серверными статусами таймера оставить только `active` и `stopped`;
- персональные настройки пользователя хранить отдельно от глобального состояния таймеров.

- `last_run_by` должен заполняться текущим пользователем;
- `time_shift_seconds` должен сохраняться отдельно от базовой длительности;
- при остановке таймера меняется только серверный статус на `stopped`.

Оценка: 1.5–3 дня.

---

### Этап 5. Подключить frontend к API

После появления API нужно заменить mock-реализацию в сервисах на реальные HTTP-запросы.

Последовательность подключения:

1. `GET /timers`
2. `POST /timers/{timer_id}/restart`
3. `POST /timers/{timer_id}/stop`
4. `POST /timers`
5. `PATCH /timers/{timer_id}`
6. `DELETE /timers/{timer_id}`

Pinia store должен продолжить отвечать за frontend-состояние, но источник данных должен перейти из моков в API.

Оценка: 1–2 дня.

---

### Этап 6. Реализовать авторизацию и роли

Нужно подключить Supabase Auth и заменить моковую роль из `usePermissions.ts`.

Минимальный MVP:

- вход по email/password;
- получение текущего пользователя;
- получение профиля пользователя;
- роль пользователя: `admin`, `manager`, `member`;
- защита frontend routes;
- проверка прав на backend.

Таблица `profiles` должна хранить:

- id;
- auth user id;
- email;
- display name;
- role;
- active status;
- timestamps.

Важно:

- frontend-ограничения остаются для удобства UI;
- настоящая защита должна быть на backend и/или Supabase RLS.

Оценка: 1.5–3 дня.

---

### Этап 7. Перенести закрепления таймеров в БД

Сейчас закрепления хранятся в `localStorage`.

Для MVP закрепления лучше перенести в таблицу:

- `user_pinned_timers`

Поля:

- id;
- user_id;
- timer_id;
- position;
- created_at;
- updated_at.

Endpoints:

- `GET /me/pinned-timers`
- `POST /me/pinned-timers`
- `DELETE /me/pinned-timers/{timer_id}`
- `PATCH /me/pinned-timers/reorder`

Правила:

- закрепления индивидуальны для пользователя;
- закрепления не влияют на других пользователей;
- один пользователь не может закрепить один и тот же таймер дважды;
- порядок закреплённых таймеров хранится через `position`;
- при удалении таймера связанные закрепления должны удаляться каскадно или через backend-логику.

Оценка: 1–2 дня.

---

### Этап 8. Реализовать изображения таймеров

Сейчас изображения отображаются через `URL.createObjectURL(file)` и не сохраняются после перезагрузки.

Для MVP нужно использовать Supabase Storage.

Логика:

1. пользователь выбирает изображение;
2. изображение загружается в storage;
3. в таймере сохраняется постоянный `image_url`;
4. при редактировании можно заменить изображение;
5. при удалении изображения `image_url` очищается.

Оценка: 1–2 дня.

---

### Этап 9. Добавить realtime-обновления

Для MVP realtime нужен только для общих таймеров.

События:

- timer created;
- timer updated;
- timer restarted;
- timer stopped;
- timer deleted.

После события frontend должен обновлять список таймеров или применять изменение локально.

Realtime для закреплений не обязателен для MVP, потому что закрепления являются персональной настройкой пользователя.

Оценка: 1–2 дня.

---

### Этап 10. Добавить логи

Таблица:

- `timer_logs`

Поля:

- id;
- timer_id;
- user_id;
- action;
- old_status;
- new_status;
- created_at.

Логировать:

- created;
- updated;
- restarted;
- stopped;
- deleted;
- role_changed.

Не логировать:

- signal;
- completed;
- sound enabled / disabled;
- global sound enabled / disabled;
- pinned;
- unpinned;
- pinned reordered;
- search;
- filter.

Оценка: 0.5–1.5 дня.

Ручная проверка:

2. Countdown работает.

9. Остановка работает.
10. Индивидуальное отключение звука у таймера работает.
11. Общее отключение звука для всех таймеров работает.
12. Если общий звук выключен, сигнал таймеров не проигрывается.
13. Если общий звук выключить во время сигнала, звук останавливается.
14. После перезагрузки страницы настройки звука сохраняются.
15. Добавление таймера работает.
16. Редактирование таймера работает.
17. Валидация длительности таймера при добавлении и редактировании работает: 0 и 1441 минут не сохраняются, 1 и 1440 минут сохраняются.
18. Удаление изображения таймера в режиме редактирования работает.
19. Удаление таймера работает.
20. Закрепление таймера работает.
21. Открепление таймера работает.
22. Закреплённые таймеры отображаются сверху.
23. Закреплённые таймеры сохраняются после перезагрузки страницы.
24. Кнопка “Изменить порядок” disabled, если закреплено меньше двух таймеров.
25. Режим перестановки закреплённых таймеров работает.
26. В reorder mode отображаются только закреплённые таймеры.
27. В reorder mode pin-кнопки скрыты, а первая колонка показывает drag handle.
28. После сохранения порядка закреплённых порядок сохраняется после перезагрузки страницы.

30. Роли admin / manager / member корректно меняют доступность глобальных действий.
31. Все роли могут закреплять таймеры и менять порядок закреплённых.
32. Esc закрывает открытые модальные окна.
33. Enter запускает таймер в модалке перезапуска, если форма валидна.
34. После открытия и закрытия модалок не остаётся лишняя фокусная обводка на кнопках строк.
