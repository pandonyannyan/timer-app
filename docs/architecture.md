# Timer App — Architecture

## Назначение

Timer App — веб-приложение для командной работы с таймерами.

Пользователи работают с общим списком таймеров, отслеживают обратный отсчёт, получают звуковые уведомления и выполняют действия в зависимости от роли.

Приложение рассчитано на небольшую команду и минимальную инфраструктуру.

## Модель таймера

Текущая frontend-модель:

```ts
export type TimerStatus = 'active' | 'stopped'

export type TimerViewStatus =
  | 'active'
  | 'warning'
  | 'stopped'
  | 'signal'
  | 'completed'

export interface Timer {
  id: string
  title: string
  description: string
  imageUrl?: string
  durationSeconds: number
  minDurationSeconds: number | null
  timeShiftSeconds: number
  startedAt: string
  lastRunBy: string
  status: TimerStatus
  soundEnabled: boolean
  createdAt: string
  updatedAt: string
}

export interface TimerFormPayload {
  title: string
  description: string
  durationMinutes: number
  minDurationMinutes: number | null
  imageFile: File | null
  removeImage?: boolean
}
```

### Основные поля

| Поле | Назначение |
|---|---|
| `durationSeconds` | Полная длительность таймера |
| `minDurationSeconds` | Опциональная минимальная длительность для warning-состояния |
| `timeShiftSeconds` | Сдвиг при запуске или перезапуске |
| `startedAt` | Время последнего запуска или перезапуска |
| `lastRunBy` | Пользователь, который последним запустил или перезапустил таймер |
| `status` | Базовый серверный статус |
| `soundEnabled` | Временное frontend-only поле для звука конкретного таймера |

`durationSeconds` используется для финального сигнала.

`minDurationSeconds` используется только для промежуточного warning-состояния.

## Статусы

### Серверные статусы

В БД должны храниться только базовые статусы:

| Статус | Значение |
|---|---|
| `active` | Таймер запущен |
| `stopped` | Таймер остановлен вручную |

### Frontend view-статусы

Frontend рассчитывает дополнительные статусы отображения:

| View-статус | Значение |
|---|---|
| `active` | Таймер идёт, минимальная и полная длительность ещё не достигнуты |
| `warning` | Минимальная длительность достигнута, полная ещё нет |
| `stopped` | Таймер остановлен |
| `signal` | Полная длительность истекла после открытия страницы пользователем |
| `completed` | Таймер считается завершённым для конкретного пользователя |

`warning`, `signal` и `completed` не являются серверными статусами.

### Правила расчёта view-статуса

| Условие | View-статус |
|---|---|
| `status = stopped` | `stopped` |
| `status = active`, `remaining <= 0`, пользователь видел окончание | `signal` |
| `status = active`, `remaining <= 0`, окончание было до открытия страницы | `completed` |
| `status = active`, запуск локально завершён пользователем | `completed` |
| `status = active`, `minDurationSeconds` достигнут, но `durationSeconds` ещё нет | `warning` |
| `status = active`, остальные случаи | `active` |

Локальный `completed` хранится по ключу:

```txt
timerId + startedAt
```

Это нужно, чтобы завершение старого запуска не применялось после перезапуска того же таймера.

## Расчёт времени

Эффективная полная длительность:

```txt
effectiveDuration = max(0, durationSeconds - timeShiftSeconds)
```

Эффективная минимальная длительность:

```txt
effectiveMinDuration =
  minDurationSeconds === null
    ? null
    : max(0, minDurationSeconds - timeShiftSeconds)
```

Прошедшее время:

```txt
elapsed = now - startedAt
```

Оставшееся время:

```txt
remaining = effectiveDuration - elapsed
```

Если таймер остановлен, в UI отображается:

```txt
00:00:00
```

## Длительность и диапазоны

Таймер может иметь:

- только полную длительность;
- минимальную и полную длительность.

Примеры ввода:

| Ввод | Значение |
|---|---|
| `100` | Полная длительность 100 минут |
| `80-100` | Минимальная длительность 80 минут, полная 100 минут |
| `80 - 100` | То же самое |

Правила валидации:

- длительность обязательна;
- значения должны быть целыми минутами;
- полная длительность должна быть от 1 до 1440 минут;
- минимальная длительность опциональна;
- если минимальная длительность указана, она должна быть меньше полной;
- `100-100` и `120-100` невалидны.

Отображение в таблице:

| Тип таймера | Отображение |
|---|---|
| Без диапазона | `100 мин` |
| С диапазоном | `80-100 мин` |

## Звуки

Звук зависит от двух пользовательских настроек:

```txt
globalSoundEnabled && timerSoundEnabled
```

Если общий звук выключен, звуки не проигрываются ни у одного таймера.

### Warning-звук

Условия:

- `viewStatus = warning`;
- звук включён глобально;
- звук включён для конкретного таймера;
- warning-звук ещё не проигрывался для текущего запуска.

Поведение:

- короткий звук проигрывается один раз;
- таймер продолжает идти до полной длительности;
- warning не логируется.

### Signal-звук

Условия:

- `viewStatus = signal`;
- звук включён глобально;
- звук включён для конкретного таймера;
- signal-звук ещё не проигрывался для текущего запуска;
- запуск не завершён локально.

Поведение:

- финальный звук проигрывается один раз;
- появляется кнопка “Завершить”;
- после нажатия “Завершить” запуск считается локально завершённым.

Звуки останавливаются при:

- остановке таймера;
- перезапуске;
- удалении таймера;
- завершении signal-состояния;
- выключении звука конкретного таймера;
- выключении общего звука.

## Перезапуск со сдвигом

В модальном окне перезапуска есть два режима:

- запустить от текущего времени;
- запустить со сдвигом.

Сдвиг вводится в минутах.

Правила:

- значение должно быть целым числом;
- значение должно быть не меньше 1 минуты;
- значение не должно превышать полную длительность таймера.

При перезапуске:

- `status = active`;
- `startedAt = now`;
- `lastRunBy = current user`;
- `timeShiftSeconds = shiftMinutes * 60`;
- `updatedAt = now`;
- локальный `completed` для таймера очищается;
- warning/signal-звуки для нового запуска могут проигрываться заново.

Сдвиг учитывается и для полной, и для минимальной длительности.

## Роли и права

Планируемые роли:

| Роль | Доступ |
|---|---|
| `admin` | Все глобальные действия с таймерами и управление пользователями |
| `manager` | Все глобальные действия с таймерами |
| `member` | Запуск и остановка таймеров |

Глобальные действия:

| Действие | admin | manager | member |
|---|---:|---:|---:|
| Добавить таймер | да | да | нет |
| Редактировать таймер | да | да | нет |
| Удалить таймер | да | да | нет |
| Перезапустить таймер | да | да | да |
| Остановить таймер | да | да | да |
| Управлять пользователями | да | нет | нет |

Персональные действия не ограничиваются ролью:

- поиск;
- фильтры;
- включение и выключение звука;
- закрепление таймеров;
- изменение порядка закреплённых таймеров.

Сейчас роли реализованы только на frontend-уровне через `usePermissions.ts`.

Позже проверки должны быть продублированы на backend и в Supabase RLS.

## Закрепления таймеров

Закрепления являются персональной пользовательской настройкой.

Текущее frontend-состояние:

```ts
pinnedTimerIds: string[]
```

Правила:

- закреплённые таймеры отображаются сверху;
- порядок закреплённых соответствует порядку id в `pinnedTimerIds`;
- закрепления не влияют на других пользователей;
- закрепить и открепить таймер может любой пользователь;
- порядок закреплённых таймеров может менять любой пользователь;
- при удалении таймера его id удаляется из `pinnedTimerIds`.

В будущем закрепления должны храниться в таблице `user_pinned_timers`.

---

## Порядок отображения

Глобального ручного порядка таймеров нет.

Список строится из двух частей:

1. закреплённые таймеры пользователя;
2. незакреплённые таймеры.

Закреплённые таймеры:

- всегда сверху;
- сортируются по пользовательскому порядку;
- отображаются независимо от статуса, если проходят поиск и фильтр.

Незакреплённые таймеры сортируются по view-статусу:

1. `signal`
2. `warning`
3. `active`
4. `stopped`
5. `completed`

Внутри одного view-статуса сортировка идёт по оставшемуся времени.

Если время одинаковое, используется сортировка по названию.

## Планируемая база данных

### `timers`

Поля:

- `id`
- `title`
- `description`
- `image_url`
- `duration_seconds`
- `min_duration_seconds`
- `started_at`
- `last_run_by`
- `time_shift_seconds`
- `status`
- `created_by`
- `updated_by`
- `created_at`
- `updated_at`

Ограничения:

```sql
check (duration_seconds > 0)
check (duration_seconds <= 86400)
check (
  min_duration_seconds is null
  or (
    min_duration_seconds > 0
    and min_duration_seconds < duration_seconds
  )
)
```

`status` должен принимать только:

- `active`
- `stopped`

`sound_enabled` не должен быть глобальным полем таймера. Звук является пользовательской настройкой.

### `user_pinned_timers`

Поля:

- `id`
- `user_id`
- `timer_id`
- `position`
- `created_at`
- `updated_at`

Ограничения:

- `unique(user_id, timer_id)`
- `unique(user_id, position)`

При удалении таймера связанные записи должны удаляться каскадно или через backend-логику.

### `user_settings`

Поля:

- `id`
- `user_id`
- `global_sound_enabled`
- `created_at`
- `updated_at`

Если записи нет, общий звук считается включённым.

### `timer_user_settings`

Поля:

- `id`
- `timer_id`
- `user_id`
- `sound_enabled`
- `created_at`
- `updated_at`

Если записи нет, звук конкретного таймера считается включённым.

### `timer_logs`

Поля:

- `id`
- `timer_id`
- `user_id`
- `action`
- `old_status`
- `new_status`
- `created_at`

Логировать:

- `created`
- `updated`
- `restarted`
- `stopped`
- `deleted`
- `role_changed`

Не логировать:

- `warning`
- `signal`
- `completed`
- включение и выключение звука;
- закрепление и открепление;
- изменение порядка закреплённых;
- поиск;
- фильтры.

## Realtime

Realtime нужен для общих таймеров.

События:

- `timer_created`
- `timer_updated`
- `timer_restarted`
- `timer_stopped`
- `timer_deleted`

После события frontend должен обновлять список таймеров.

Realtime для закреплений не обязателен для MVP, потому что закрепления являются персональной настройкой пользователя.

## Ограничения текущего этапа

- нет backend;
- нет авторизации;
- нет настоящих пользователей;
- нет синхронизации между пользователями;
- роли моковые;
- таймеры работают через Pinia и mock-реализацию `timerService`;
- изображения не сохраняются после перезагрузки;
- настройки звука хранятся в `localStorage`;
- закрепления хранятся в `localStorage`;
- изменения таймеров не сохраняются после перезагрузки.

## Принципы

- сервер хранит только базовое состояние таймера;
- frontend рассчитывает view-статусы;
- `warning`, `signal` и `completed` не являются серверными статусами;
- локальный `completed` относится к конкретному запуску таймера;
- полная длительность отделена от минимальной длительности;
- параметры запуска отделены от базовой длительности;
- порядок таймеров не является свойством таймера;
- закрепления и настройки звука являются пользовательскими настройками;
- frontend-ограничения не считаются безопасностью;
- роли должны проверяться на frontend и backend.
```
