# Timer App — Architecture

## Общее описание

Timer App — веб-приложение для командной работы с таймерами.

Пользователи видят общий список таймеров, отслеживают обратный отсчёт, получают звуковые уведомления по истечении времени и могут выполнять действия с таймерами в зависимости от своей роли.

Каждый пользователь может персонально закреплять важные для себя таймеры и менять порядок закреплённых таймеров. Эти настройки не влияют на других пользователей.

Приложение рассчитано на небольшую команду и строится с учётом бесплатной инфраструктуры.

---

## Страницы приложения

Планируемые страницы:

- /
- /logs
- /admin/users

Назначение:

- / — список таймеров;
- /logs — лог действий;
- /admin/users — управление пользователями.

На текущем этапе реализована основная страница со списком таймеров.

## Основная модель таймера

Текущая frontend-модель:

export type TimerStatus = 'active' | 'stopped'

export type TimerViewStatus = 'active' | 'stopped' | 'signal' | 'completed'

export interface Timer {
  id: string
  title: string
  description: string
  imageUrl?: string
  durationSeconds: number
  timeShiftSeconds: number
  startedAt: string
  lastRunBy: string
  status: TimerStatus
  soundEnabled: boolean
  createdAt: string
  updatedAt: string
}

export interface TimerFormPayload {
  title: string
  description: string
  durationMinutes: number
  imageFile: File | null
  removeImage?: boolean
}

### Важные поля

- startedAt — время последнего запуска или перезапуска таймера;
- lastRunBy — пользователь, который последним запустил или перезапустил таймер;
- durationSeconds — базовая длительность таймера;
- timeShiftSeconds — сдвиг при запуске или перезапуске;
- status — базовый серверный статус таймера;
- soundEnabled — временное frontend-only поле; в будущем звук должен храниться как пользовательская настройка.

## Статусы таймера

### Серверные статусы

В будущем в БД должны храниться только базовые серверные статусы:

- active
- stopped

### Frontend view-статусы

На фронте дополнительно рассчитываются view-статусы:

- active
- stopped
- signal
- completed

### Поведение статусов

| Условие | Отображение |
|---|---|
| status = active и remaining > 0 | active |
| status = stopped | stopped |
| status = active, remaining <= 0, пользователь видел окончание | signal |
| status = active, remaining <= 0, пользователь открыл страницу позже | completed |
| пользователь нажал “Завершить” | completed локально |

signal и completed не являются полноценными серверными статусами. Это локальное отображение состояния таймера для конкретного пользователя.

---

## Логика статусов

### active

Таймер запущен и ещё не истёк.

### stopped

Таймер остановлен вручную.

### signal

Таймер истёк уже после открытия страницы пользователем.

Используется для сценария:

- пользователь видит окончание таймера;
- проигрывается звук;
- появляется кнопка “Завершить”.

### completed

Таймер считается завершённым для конкретного пользователя.

Сценарии:

- таймер истёк до открытия страницы;
- пользователь нажал “Завершить”;
- таймер находится в локальном completed-хранилище.

## Расчёт времени

При запуске или перезапуске таймера может использоваться сдвиг:

timeShiftSeconds

Эффективная длительность:

effectiveDuration = durationSeconds - timeShiftSeconds

Оставшееся время:

remaining = effectiveDuration - elapsed

Где:

elapsed = now - startedAt

Если таймер остановлен, отображается:

00:00:00

---

## Перезапуск таймера со сдвигом

В модальном окне перезапуска есть два режима:

- запустить от текущего времени;
- запустить со сдвигом.

Сдвиг вводится в минутах.

Ограничения:

- shiftMinutes должно быть целым числом;
- shiftMinutes >= 1;
- shiftMinutes не должно превышать базовую длительность таймера.

При перезапуске:

- status = active;
- startedAt = now;
- lastRunBy = current user;
- timeShiftSeconds = shiftMinutes * 60;
- updatedAt = now.

На текущем frontend-only этапе lastRunBy заполняется моковым пользователем. После подключения авторизации это поле должно заполняться данными текущего пользователя.

Также сбрасывается локальный completed для этого таймера.

## LocalStorage

LocalStorage используется для персональных локальных состояний пользователя.

### Локально завершённые таймеры

Используется для сценария “Завершить”.

Если пользователь нажал “Завершить” у таймера в состоянии signal, id таймера сохраняется локально.

При следующем расчёте такой таймер отображается как completed для этого пользователя.

При перезапуске или удалении таймера локальный completed очищается.

### Настройки звука

Отключение звука хранится персонально для пользователя.

Сейчас есть два уровня настроек:

- общий звук для всех таймеров;
- звук конкретного таймера.

Итоговое правило:

globalSoundEnabled && timerSoundEnabled

Звук проигрывается только если:

- общий звук включён;
- звук конкретного таймера включён.

Если общий звук выключен, сигналы не проигрываются ни у одного таймера, даже если у конкретного таймера индивидуальный звук включён.

Если общий звук выключить во время проигрывания сигнала, текущий звук останавливается.

Пока настройки хранятся в localStorage. В будущем они должны храниться в БД как пользовательские настройки.

### Закрепления таймеров

Закрепления таймеров сейчас хранятся в localStorage.

Используется composable:

src/composables/usePinnedTimers.ts

Основное состояние:

pinnedTimerIds: string[]

Порядок id в массиве является пользовательским порядком закреплённых таймеров.

Правила:

- закрепление таймера является персональной настройкой пользователя;
- закрепления не влияют на других пользователей;
- закрепить и открепить таймер может любой пользователь;
- порядок закреплённых таймеров может менять любой пользователь;
- при удалении таймера его id удаляется из pinnedTimerIds.

В будущем закрепления должны храниться в БД в таблице user_pinned_timers.

---

## Звуковое уведомление

Условия воспроизведения звука:

- status = active;
- remaining <= 0;
- viewStatus = signal;
- globalSoundEnabled = true;
- timerSoundEnabled = true;
- звук ещё не проигрывался;
- таймер не завершён локально.

Эффективное состояние звука рассчитывается так:

const effectiveSoundEnabled = globalSoundEnabled && timerSoundEnabled

Звук останавливается при:

- завершении;
- остановке;
- перезапуске;
- удалении таймера;
- отключении звука конкретного таймера;
- отключении общего звука.

## Роли пользователей

Планируемые роли:

| Роль | Доступ |
|---|---|
| admin | Все глобальные действия с таймерами + управление пользователями |
| manager | Все глобальные действия с таймерами, кроме управления пользователями |
| member | Запуск и остановка таймеров |

### Права по глобальным действиям

| Действие | admin | manager | member |
|---|---:|---:|---:|
| Добавить таймер | да | да | нет |
| Редактировать таймер | да | да | нет |
| Удалить таймер | да | да | нет |
| Перезапустить таймер | да | да | да |
| Остановить таймер | да | да | да |
| Управлять пользователями | да | нет | нет |

### Персональные действия пользователя

Не ограничиваются ролью:

- включение/выключение звука;
- поиск;
- фильтры;
- закрепление таймеров;
- открепление таймеров;
- изменение порядка закреплённых таймеров.

Эти действия считаются локальными или персональными для пользователя и не меняют глобальную структуру таймеров.

Сейчас роли реализованы только на frontend-уровне через usePermissions.ts. Позже ограничения для глобальных действий должны быть продублированы на backend/Supabase RLS.

## Действия с таймерами

| Действие | Условие | Результат |
|---|---|---|
| Добавить | admin, manager | создаётся новый таймер в Pinia |
| Редактировать | admin, manager | обновляются данные таймера в Pinia |
| Удалить | admin, manager | таймер удаляется из Pinia |
| Перезапустить | все роли | status = active, startedAt = now, обновляются lastRunBy и timeShiftSeconds |
| Остановить | все роли, только active | status = stopped |
| Завершить | signal | локально переводит таймер в completed |
| Выключить звук таймера | все роли | сохраняется индивидуальная настройка конкретного таймера |
| Выключить общий звук | все роли | сохраняется общая пользовательская настройка звука |
| Закрепить таймер | все роли | таймер добавляется в пользовательский pinnedTimerIds |
| Открепить таймер | все роли | таймер удаляется из пользовательского pinnedTimerIds |
| Изменить порядок закреплённых | все роли, если закреплено больше одного таймера | меняется пользовательский порядок закреплённых таймеров |

## Управление модальными окнами с клавиатуры

Общее правило:

- Esc закрывает открытую модалку;
- после открытия и закрытия модалки фокусная обводка не должна оставаться на кнопке, которая открыла модалку.

Для модалки перезапуска дополнительно:

- Enter запускает таймер, если текущее состояние формы валидно;
- если выбран запуск со сдвигом и значение некорректно, Enter не выполняет запуск.

Для модалки удаления:

- Enter не подтверждает удаление, чтобы избежать случайного удаления таймера.

## Добавление и редактирование таймера

Для добавления и редактирования используется один компонент модального окна.

### Добавление

Поля:

- название;
- описание;
- длительность в минутах;
- изображение.

После сохранения создаётся новый таймер в Pinia.

### Редактирование

Открывается та же модалка, но в режиме редактирования.

Поля заполняются текущими значениями таймера.

Можно изменить:

- название;
- описание;
- длительность;
- изображение;
- удалить текущее изображение.

Если новое изображение не выбрано и удаление не отмечено, старое изображение сохраняется.

Если пользователь нажимает кнопку удаления изображения, текущее изображение помечается на удаление. Реальное удаление применяется только после нажатия “Сохранить”. Если закрыть модалку через “Отмена”, изображение останется без изменений.

### Валидация

- название обязательно;
- длительность обязательна;
- длительность должна быть целым числом минут;
- длительность должна быть больше 0.

## Изображения таймеров

Сейчас выбранное изображение отображается через:

URL.createObjectURL(file)

Это временное frontend-only решение.

Ограничения:

- картинка работает только в текущей сессии браузера;
- после перезагрузки страницы она не сохранится;
- в будущем изображение нужно загружать в Supabase Storage;
- в таймере нужно хранить постоянный imageUrl.

В режиме редактирования:

- если у таймера уже есть изображение, рядом с полем выбора файла показывается кнопка удаления;
- кнопка удаления использует общий стиль IconButton и иконку удаления;
- кнопка удаления расположена по правому краю строки выбора файла;
- удаление изображения передаётся через removeImage: true;
- при сохранении updateTimer очищает imageUrl;
- если после удаления выбрать новый файл, сохраняется новый файл.

---

## Удаление таймера

Удаление происходит через окно подтверждения.

Сценарий:

1. Пользователь нажимает кнопку удаления.
2. Открывается ConfirmDeleteModal.
3. Кнопка “Отмена” закрывает окно.
4. Кнопка “Удалить” удаляет таймер из Pinia.
5. Локальный completed для таймера очищается.
6. Локальное закрепление таймера очищается.

---

## Порядок отображения таймеров

Глобальный ручной порядок таймеров удалён.

Порядок списка формируется на frontend-уровне из двух частей:

1. закреплённые таймеры пользователя;
2. незакреплённые таймеры.

### Закреплённые таймеры

Закреплённые таймеры:

- всегда отображаются сверху;
- отображаются независимо от статуса;
- сортируются по пользовательскому порядку закрепления;
- не влияют на порядок у других пользователей.

Порядок закреплённых таймеров хранится в массиве pinnedTimerIds.

Пример:

pinnedTimerIds = ['timer-3', 'timer-1', 'timer-8']

Это означает:

1. timer-3;
2. timer-1;
3. timer-8.

### Незакреплённые таймеры

Незакреплённые таймеры отображаются автоматически по view-статусу:

1. signal
2. active
3. stopped
4. completed

Внутри одного view-статуса таймеры сортируются по оставшемуся времени.

Если оставшееся время одинаковое, используется сортировка по названию как fallback, чтобы порядок был стабильнее.

---

## Перестановка закреплённых таймеров

Реализована drag-and-drop перестановка закреплённых таймеров.

### Сценарий

1. Пользователь закрепляет два или больше таймеров.
2. Кнопка “Изменить порядок” становится активной.
3. Пользователь нажимает “Изменить порядок”.
4. Таблица переходит в reorder mode.
5. В reorder mode отображаются только закреплённые таймеры.
6. Поиск, фильтры и действия в строках становятся disabled.
7. Первая колонка показывает drag handle.
8. Pin-кнопки скрыты.
9. Пользователь перетаскивает строки drag-and-drop.
10. Новый порядок хранится во временном draft-состоянии.
11. После нажатия “Сохранить порядок” обновляется pinnedTimerIds.
12. Таблица возвращается в обычный режим.

Drag-and-drop реализован без сторонних библиотек.

## Поиск и фильтры

### Поиск

Поиск работает по:

- названию;
- описанию.

Поиск применяется и к закреплённым, и к незакреплённым таймерам.

Если закреплённый таймер не подходит под поисковый запрос, он не отображается в текущем результате.

### Фильтр

Доступные фильтры:

- все;
- активные.

Фильтр “Активные” показывает:

- active;
- signal.

Фильтр применяется и к закреплённым, и к незакреплённым таймерам.

Если закреплённый таймер не подходит под фильтр, он не отображается в текущем результате.

### Сортировка

Ручная сортировка по колонкам удалена.

Порядок списка задаётся автоматически:

- закреплённые сверху в пользовательском порядке;
- незакреплённые ниже в порядке signal / active / stopped / completed.

## Pinia store

Файл:

src/stores/timers.ts

Сейчас store содержит мок-данные и actions.

### Actions

- createTimer(payload)
- updateTimer(timerId, payload)
- deleteTimer(timerId)
- restartTimer(timerId, timeShiftSeconds)
- stopTimer(timerId)

### createTimer

Создаёт новый таймер локально.

Особенности:

- генерирует id;
- задаёт createdAt, updatedAt, startedAt;
- статус нового таймера — active;
- timeShiftSeconds = 0;
- soundEnabled = true;
- таймер добавляется в локальный массив.

Позиция таймера в массиве не является продуктовым порядком отображения.

### updateTimer

Редактирует существующий таймер локально.

Обновляет:

- title;
- description;
- durationSeconds;
- imageUrl, если выбран новый файл;
- очищает imageUrl, если передано removeImage: true;
- updatedAt.

Не меняет:

- startedAt;
- status;
- timeShiftSeconds.

### deleteTimer

Удаляет таймер из локального массива.

Также:

- очищает локальный completed через removeCompletedTimer(timerId).

Локальное закрепление таймера очищается на уровне UI/composable, чтобы store таймеров не смешивался с пользовательскими настройками.

### restartTimer

Перезапускает таймер:

- status = active;
- startedAt = now;
- lastRunBy = current user;
- timeShiftSeconds = переданное значение;
- updatedAt = now.

Также:

- очищает локальный completed.

### stopTimer

Останавливает таймер:

- status = stopped;
- updatedAt = now.

## Пользовательские закрепления

Файл:

src/composables/usePinnedTimers.ts

Сейчас composable работает с localStorage.

### Состояние

pinnedTimerIds: string[]

Порядок id в массиве является пользовательским порядком закреплённых таймеров.

### Методы

- isTimerPinned(timerId)
- pinTimer(timerId)
- unpinTimer(timerId)
- togglePinnedTimer(timerId)
- reorderPinnedTimers(timerIds)
- removePinnedTimer(timerId)

### Правила

- если timerId уже закреплён, повторное закрепление не дублирует его;
- при закреплении timerId добавляется в конец pinnedTimerIds;
- при откреплении timerId удаляется из pinnedTimerIds;
- при reorder массив pinnedTimerIds заменяется новым порядком;
- при удалении таймера его id удаляется из pinnedTimerIds.

## Планируемая таблица timers

Предполагаемая структура в БД:

- id
- title
- description
- image_url
- duration_seconds
- started_at
- last_run_by
- time_shift_seconds
- status
- created_by
- updated_by
- created_at
- updated_at

started_at — время последнего запуска или перезапуска таймера. Используется для расчёта оставшегося времени и для отображения даты/времени последнего запуска.

Важно:

- last_run_by в будущем должен ссылаться на пользователя, который последним запустил или перезапустил таймер;
- sound_enabled, скорее всего, не должен быть глобальным полем таймера;
- отключение звука сейчас локально для пользователя;
- в будущем это может быть отдельная user settings таблица;
- поле глобального порядка в таблице timers не планируется.

---

## Планируемая таблица user_pinned_timers

Закрепления таймеров должны храниться отдельно от таблицы timers.

Поля:

- id
- user_id
- timer_id
- position
- created_at
- updated_at

Правила:

- закрепление индивидуально для каждого пользователя;
- один пользователь не может закрепить один и тот же таймер дважды;
- position задаёт порядок закреплённых таймеров пользователя;
- если записи для таймера нет, таймер считается незакреплённым для этого пользователя;
- удаление записи открепляет таймер;
- изменение position меняет порядок закреплённых таймеров.

Желательные ограничения:

- unique(user_id, timer_id)
- unique(user_id, position)

При удалении таймера связанные записи user_pinned_timers должны удаляться каскадно или через backend-логику.

## Пользовательские настройки

В будущем пользовательские настройки лучше хранить отдельно от таймеров.

### user_settings

Глобальные настройки пользователя.

Поля:

- id
- user_id
- global_sound_enabled
- created_at
- updated_at

Правила:

- если записи нет, общий звук считается включённым;
- настройка индивидуальна для каждого пользователя;
- настройка не влияет на других пользователей.

### timer_user_settings

Настройки пользователя для конкретного таймера.

Поля:

- id
- timer_id
- user_id
- sound_enabled
- created_at
- updated_at

Правила:

- если записи нет, звук конкретного таймера считается включённым;
- настройка индивидуальна для каждого пользователя и таймера;
- настройки звука не логируются.

Итоговое правило проигрывания звука:

globalSoundEnabled && timerSoundEnabled

### user_pinned_timers

Порядок закреплённых таймеров пользователя.

Поля:

- id
- user_id
- timer_id
- position
- created_at
- updated_at

Правила:

- если записи нет, таймер не закреплён для пользователя;
- закрепление индивидуально для пользователя;
- position задаёт пользовательский порядок закреплённых таймеров;
- закрепления не логируются как действия таймера.

## Realtime

В будущем frontend должен подписываться на изменения таймеров через Supabase Realtime.

События таймеров:

- timer_created
- timer_updated
- timer_restarted
- timer_stopped
- timer_deleted

После события frontend должен обновлять список таймеров.

Закрепления пользователя являются персональной настройкой. Их можно загружать отдельно при входе пользователя и обновлять после успешного сохранения порядка. Realtime для закреплений не обязателен для MVP.

- нет backend;
- нет авторизации;

## Принципы

- минимальная сложность;
- понятная frontend-архитектура;
- отделение серверного состояния от локального пользовательского состояния;
- базовая длительность таймера отделена от параметров запуска;
- timeShiftSeconds хранится отдельно;
- startedAt является временем последнего запуска или перезапуска;
- порядок таймеров не является глобальным свойством таймера;
- закрепления и порядок закреплённых таймеров являются пользовательской настройкой;
- закреплённые таймеры всегда отображаются сверху;
- незакреплённые таймеры сортируются автоматически по view-статусу;
- поиск, фильтры, настройки звука и закрепления — локальны или персональны для пользователя;
- звук имеет два уровня: общий пользовательский переключатель и индивидуальную настройку конкретного таймера;
- роли должны быть проверены на frontend и backend;
- frontend-ограничения не считаются безопасностью.
